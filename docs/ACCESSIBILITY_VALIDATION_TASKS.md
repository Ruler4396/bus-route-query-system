# PJT-0001 · ACCESSIBILITY_VALIDATION_TASKS

最后更新：2026-03-07  
验证类型：**准真实用户验证（脚本代理 + 试点演示环境）**

## 1. 目的
对当前系统做一轮**可复现的准真实任务验证**，验证重点不是“系统已经完全成熟”，而是：
- 关键任务是否可完成
- 用户是否能理解推荐和风险提示
- 系统是否会明显误导用户
- 当前哪些地方仍然需要继续迭代

## 2. 验证边界
- 环境：`8.134.206.52:8134` 演示实例
- 数据：H2 演示数据 + 试点范围样本
- 主人群：轮椅 / 行动不便、低视力
- 验证方式：Playwright 准真实任务脚本代理
- 非覆盖项：真实屏幕阅读器、人类现场试走、线下人工核验

## 3. 任务清单
### T01 · 轮椅画像路线选择任务
- 目标：从多条候选路线中识别适合轮椅 / 行动不便人群的路线
- 操作：进入路线规划页，选择 `WHEELCHAIR` 画像，输入模糊起终点 `总站 -> 总站`，执行无障碍推荐
- 通过标准：
  - 推荐摘要卡出现
  - 第一条推荐为 `1路`
  - 至少有一条关键数据不足的路线被过滤或降级

### T02 · 低视力画像路线选择任务
- 目标：验证不同画像会得到不同结果，而不是换标签不换结果
- 操作：同样输入 `总站 -> 总站`，切换到 `LOW_VISION`
- 通过标准：
  - 推荐摘要卡出现
  - 第一条推荐为 `3路`
  - 推荐卡中能看到 `置信度 / 状态 / 数据源`

### T03 · 分段风险理解任务
- 目标：验证用户是否能理解“从出发到到达”的分段说明
- 操作：查看推荐卡中的分段清单
- 通过标准：
  - 能看到 `出发步行段 / 上车站可达性 / 公交乘车段 / 换乘设施评估 / 下车站可达性 / 到达步行段`
  - 至少一段有明确风险或谨慎核对状态

### T04 · 数据治理理解任务
- 目标：验证用户能否理解系统不是拍脑袋推荐，而是有数据来源与置信度规则支撑
- 操作：查看路线规划页治理面板
- 通过标准：
  - 能看到 `数据源登记`
  - 能看到 `置信度规则`
  - 能看到 `试点人工样本基础`

### T05 · 反馈闭环任务
- 目标：验证用户发现问题后，系统是否有闭环而不是只能停留在“抱怨”
- 操作：进入留言建议页 -> 进入反馈处理看板 -> 更新状态
- 通过标准：
  - 页面能看到结构化反馈字段（类型、严重级别、路线、站点）
  - 处理看板可修改状态并保存

## 4. 记录指标
每个任务至少记录：
- 是否完成
- 耗时（秒）
- 关键页面
- 是否出现误导风险
- 失败点 / 疑点
- 理解成本（低 / 中 / 高）

## 5. 当前结论门槛
如果本轮准真实任务验证中：
- 任务完成率 >= 80%
- 没有出现“明显误导用户”的致命问题
- 失败点都可被清晰记录和追溯
则可以把 H01 / H02 / H03 视为完成，但**不等于真实用户验证已经完成**。
